I - Introduction

3/ En résumé

  • A l'intérieur de la BDD , les informations sont classées, structurées et regroupées par sujet.
  • Le langage de requête SQL nous permet d'échanger les informations avec la base; autrement dit il est utilisé pour la communication entre le client et le serveur.
  • Dans une BDD , les informations sont présentées sous forme de tables.
  • Dans une table, nous avons des colonnes , et chaque donnée y est introduite sous forme de ligne.
  • Une fois, les informations enrégistrées , il est important de les gérer (ajout, modification, suppression, consultation).

II- Modélisation

  • Avant de créer une BDD, il est primordial de réfléchir à la modélisation de la base: afin d'éviter de se tromper.
  • Changer la modélisation d'une BDD en cours de projet n'est pas recommandé. Cela induit obligatoirement une réecriture des requêtes

    • Comment modéliser une BDD ?

    On peut modéliser une BDD sur une feuille de papier ou on pourrait utiliser un logiciel. Exemple : MySQL workbench

    Pour une bonne conception de BDD, il faut réfléchir en terme de sujet en modélisant ceux du monde réel. La modélisation se compose de différentes tables (une table = à un sujet) ; ex. si nous vendons des produits ; il faut une table des produits et une table des commandes.

III- Les tables

  • 1 sujet représente 1 table dans une base de données.
  • 1 table est un emplacement de sauvegarde.
  • Les données sont stockées à l'intérieur de tables. Une table peut être comparée à une liste, qui contient des enregistrements relatifs à un sujet bien défini.

Il faut réfléchir au sujet et à ses conséquences. Si nous vendons des produits, nous aurons certainement des membres. 1 table membre sera donc nécessaire. En terme de fonctionnalité, si nous souhaitons proposer un abonnement newsletter aux membres, nous aurons besoin d'1 table newsletter repertoriant les membres abonnés
table = 1 sujet

IV- Les colonnes ou les champs

Chaque table possède généralement plusieurs champs (aussi appelées colonnes ). les colonnes/champs représentent des caractéristiques relatives au sujet (la table)

Pour savoir, quelle colonne mettre dans quelle table, il faut se poser la question suivante : qu'est-ce qui pourrait décrire mon sujet ? quelles sont les informations sur mon sujet ?
Par exemple, la table membre aura les champs : pseudo, mot de passe, nom, prenom, adresse, email, etc.

V- Les types de données

Type de champ Description Contexte
VARCHAR Chaîne (jusqu'à 256 caractères) Pour enregistrer un pseudo, un email, le titre d'un produit ou d'un article de blog
TEXT Chaîne de caractère (illimité) Pour un contenu textuel long, un article de blog
INT Numérique Pour un identifiant, un prix, un code un code postal le nombre de produits en stocks etc.
DATE Date Pour la date d'un enregistrement (une inscription), la date d'une commande etc.
ENUM Une liste (une énumération) Pour proposer une liste de termes ex. : "Madame", "Monsieur"

VI- Types de données PK AI pour les ID

A/ PK- Primary Key , clef primaire

  • Les identifiants sont des champs ou des colonnes un peu particuliers car ils ne décrivent pas le sujet.Ils sont présents dans chaque table et systématiquement en première position.
  • Les identifiants sont uniques.
  • Chaque table possède un champ identifiant , nous appelons cela une clef primaire.
  • Par ex. nous pourrons appeler le champ "id" ou encore "id_membre" ou "idMembre" "id_user" pour la table "membre" ou "user"

B/ Auto Increment , auto incrémenté

  • Auto incrémenté : option qui permet de générer un numéro unique dans une colonne ou un champ de type clef primaire(identifiant).
  • Par conséquent,retenez que le premier champ dechaque table sera systématiquement un "iD" qui sera PK et AI

C/ NULL, NOT NULL

  • Dans chaque champ nous pourrons indiquer si nous acceptons les valeurs NULL ou NOT NULL
  • NULL est un type de valeur en informatiqueévitant de laisser un champ vide si nous n'avions pas d'information à y déposer

D/ Les relations

Il arrive parfois que les sujets interagissent entre eux: exemples

  • Un mebre commande un produit ou un produit est commandé par un membre , nous devrons enrégistrer l'information dans une table
  • Un conducteur conduit un véhicule ou un véhicule est conduit par un conducteur ; il faudrait une table permettant de faire le lien entre le conducteur et le type de véhicule qu'il conduit.

VII- Les cardinalités

  • Les cardinalités servent à connaître le chiffre min et max d'enrégistrement dans une relation
  • Exemple: Un homme est le fils d'une d'une femme et une d'une seule (min =1 et max = 1)
    En revanche une femme peut n'avoir aucun enfant ou plusieurs (min =0 et max = n)
  • Exemple 2: 1 livre possède forcément 1 auteur et qu'1 seul. Relation (1,1). Minimum 1 auteur, Maximum 1 auteur. 1 auteur peut avoir écrit plusieurs livre (ou aucun). Relation (0,n). Minimum 0 livre, Maximum N livres

VIII- Les Tables de jointure

Une table de jointure permet de faire le lien entre deux tables au minimum.

Exemple: Une société de taxi , qui possède des conducteurs et des véhicules.

  • Un conducteur peut conduire 0 ou plusieurs véhicules
  • Un véhicule peut être conduit par 0 ou plusizurs conducteurs.

La table de jointure répond à la question : Comment savoir quel conducteur conduit quel véhicule ? ou Quel véhicule est conduit par quel conducteur?

Avec cette table de jointure , nous savons qu'Alexandre conduit est autorisé à conduire la BMW et la Mercedes , et Julien ne pourrait conduire uniquement que la BMW

IX- Clefs étrangères ou FK

Les clefs étrangères sont des chmpas forcément clefs primaires dans leur table d'origine et qui se retrouve dans une table extérieure.

Un champ clé étrangère permet de faire la relation avec un autre sujet.

X- Les enrégistrements

Chaque enrégistrement représente une ligne dans la table de la BDD. Chaque nouvel enrégistrement va s'autoincrémenter. Un "iD" supprimé ne sera pas réutilisé !

XI- Interroger la BDD

Les requêtes

Une requête permet de poser une question afin d'obtenir une réponse ou encore de donner un ordre.

Nous terminerons toutes nos requêtes SQL par un point-virgule (;) pour que MySQL sache que nous avons terminé d'écrire et qu'il doit exécuter notre demande.

De manière standard, les requêtes SQL seront écrites en majuscules SELECT * FROM membre;

Types de requêtes

Il existe 4 types de requêtes:

  • Sélection SELECT; requête question/réponse , nous demandons et obtenons une réponse
  • InsertionINSERT; requête d'action , qui aura un impact sur les données
  • ModificationUPDATE;requête d'action , qui aura un impact sur les données(mise à jour)
  • SuppressionDELETE;requête d'action , qui aura un impact sur les données (supprimer)
